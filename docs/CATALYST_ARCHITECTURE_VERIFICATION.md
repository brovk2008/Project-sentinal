This document contains the verified design choices, constraints, and architecture mapping for the **Project Sentinel** platform, addressing the questions in [08_Catalyst_Architecture.md](file:///C:/Users/techp/Downloads/more%20projects/Project%20Sentinel/for%20v2/08_Catalyst_Architecture.md).

## 1. AppSail WebSocket Verification & Streaming Design

### Verification
Zoho Catalyst's AppSail PaaS container environment is designed for serverless compute, running stateless applications.
*   **WebSockets Constraint**: AppSail does *not* natively support persistent, long-lived bidirectional WebSocket connections. It is optimized for HTTP request-response cycles. Instances are subject to scale-down/idle termination and timeout limits, which disconnect active WebSockets.
*   **Managed Event Bus**: While WebSockets are unsupported on AppSail containers and Catalyst Functions, Zoho offers **Catalyst Signals** to act as a managed event bus, routing real-time webhook exchanges.

### Architectural Decision
To implement real-time server-to-client streaming (e.g. streaming LLM completions or progress percentages of background ingestion jobs):
1.  **Server-Sent Events (SSE)**: For unidirectional streaming from backend to frontend, we will use SSE over HTTP, which works natively within AppSail's container gateway.
2.  **Short Polling**: For background status updates (such as RAG document index progress), the frontend will short-poll the status endpoints (e.g. `GET /api/v2/cases/{id}/memory/documents/status`) every 2 seconds.
3.  **Catalyst Signals**: If persistent bidirectional messaging is later required, we will integrate Catalyst Signals rather than hosting a standalone stateful WebSocket cluster.

## 2. File Store Naming, Offerings & Limits

### Verification
*   **Service Name**: Zoho Catalyst's managed object store is named **File Store** (sometimes referred to under the Stratus namespace in developer consoles).
*   **File Size Limit**: The maximum size of any single uploaded file is **100 MB**.
*   **Storage Limits**: 
    *   **Development**: 1 GB total storage per project.
    *   **Production**: Unlimited storage on a pay-per-use basis.
*   **Identifier**: Every uploaded file is assigned a unique **File ID** string.
*   **Bucket/Folder Naming**: Bucket/Folder names must be between 3 and 50 characters, contain only alphanumeric characters and hyphens, and cannot start or end with a hyphen.

### Architectural Decision
*   **RAG Uploads**: All PDFs, audio, video, and Shapefiles will be uploaded to Catalyst File Store.
*   **Reference Schema**: The database will not store raw files; it will store the unique `File ID` generated by the Catalyst SDK alongside metadata.
*   **Size Guards**: The frontend and backend routers will validate file uploads and reject any file exceeding the 100 MB limit with a `413 Payload Too Large` error, prompting the user to split files if necessary.

## 3. Managed User Authentication & RBAC

### Verification
Zoho Catalyst provides a managed **Authentication & User Management** service out-of-the-box.
*   **Access Control (RBAC)**: Catalyst allows the creation of **Custom Roles** (e.g. Admin, Editor, Viewer).
*   **Table Scopes**: Permissions are enforced at the Data Store table level. Each role can be configured with **Table Scopes** (Global, Org, User) and specific operation grants (Select, Insert, Update, Delete).
*   **Auth Integrations**: Supports email/password login and social OAuth logins (Google, Zoho, Facebook, LinkedIn, Microsoft).

### Architectural Decision
*   **User Registry**: We will map our RBAC model to Catalyst's native roles:
    *   `App Administrator` / `Admin` role maps to Project Sentinel **Owner**.
    *   `Editor` role maps to Project Sentinel **Editor** (can modify graph entities, upload documents).
    *   `Viewer` role maps to Project Sentinel **Viewer** (read-only access to dashboards).
*   **Data Store Scoping**: The `cases`, `entities`, and `relationships` tables in Catalyst Data Store will utilize **Org Scope** to ensure collaborative multi-user editing within an organization space, while maintaining isolation between customer tenants.

## 4. Data Store Write-Throughput & Migration Design

### Verification
*   **Rate Limits**: Zoho Catalyst enforces a standard API limit of **100 requests per minute** per developer instance. Exceeding this returns an `HTTP 429 Too Many Requests` error.
*   **Free vs. Paid**: While there is no hard-capped write-throughput ceiling on the production tier, high-frequency individual row inserts will quickly trigger 429 rate limit exceptions on standard API connections.

### Architectural Decision
Standard row-by-row imports of the 1.67M+ row dataset are structurally prohibited. To achieve maximum throughput during the migration:
1.  **Local Staging Engine**: Use local SQLite database replication for local runs and staging.
2.  **Bulk Operations**: Use Catalyst Data Store **Bulk Insert / CSV Import APIs** rather than single-record inserts.
3.  **Transactional Batching**: Perform transactions in batches of 5000 in AppSail compute before syncing back or writing to SQLite local stores to minimize API request overhead.
4.  **Throttling & Backoff**: Standardize error handlers to respect the `Retry-After` header when making API calls.
